[实现决策]
  1. SeaTunnel > DataX: SeaTunnel有原生HTTP连接器，DataX需要自己写
  2. dbt在Airflow容器内运行，不单独部署dbt服务（dbt是CLI工具，无状态）
  3. ODS用view而不是table：始终反映源表最新数据，避免ODS层数据不同步
  4. DWD/DWS用incremental：提高增量处理效率
  5. ADS用table：数据量小(聚合结果)，全量重建更可靠
  6. 单PostgreSQL双库：本地开发简化，生产应分离
  7. SeaTunnel Zeta本地模式：无需Spark/Flink集群，REST API直接触发

[已知限制和风险]
  1. GitHub API 未认证限60次/小时 → 单源DAG每30分钟调度，刚好不超
  2. 当前使用 GitHub Events API（每次30条），不是完整的 GH Archive（NDJSON格式更复杂）
  3. DummyJSON 数据是静态fake数据，不更新，新鲜度检查设为48小时
  4. SeaTunnel Zeta REST API的 curl POST 方式不够健壮，生产应改用 SimpleHttpOperator
  5. Webhook URL未配置时告警静默跳过，不会报错（可接受）
  6. 企微Markdown不支持表格中的 | 管道符，需转义
  7. docker-compose.yml 中 Fernet Key 和 Secret Key 是硬编码的，生产环境应使用 Secrets
